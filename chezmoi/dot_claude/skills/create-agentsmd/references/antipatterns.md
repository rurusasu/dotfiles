# AGENTS.md アンチパターン集

## 1. 過剰な情報量

### ❌ 悪い例

```markdown
# AGENTS.md

## プロジェクト概要

[100行にわたる詳細な説明...]

## 技術スタック

- React 18.2.0 (具体的なバージョンは不要)
- TypeScript 5.3.2
- [50行のライブラリリスト...]

## APIエンドポイント一覧

- GET /api/users - ユーザー一覧取得
- GET /api/users/:id - ユーザー詳細取得
  [100個のエンドポイント詳細...]

## データベーススキーマ

[全テーブルの詳細定義...]
```

### ✅ 良い例

```markdown
# AGENTS.md

## プロジェクト概要

ユーザー管理システムのバックエンドAPI

## 技術スタック

- FastAPI + SQLAlchemy
- PostgreSQL

## 詳細ドキュメント

- APIエンドポイント: docs/API.md
- データベーススキーマ: docs/SCHEMA.md
```

**理由**: 詳細情報は別ファイルに分離し、必要なときだけ読み込ませる（Progressive Disclosure）

---

## 2. スタイルガイドの全文コピー

### ❌ 悪い例

```markdown
## コードスタイル

- インデントは2スペース
- セミコロンは省略
- シングルクォートを使用
- 行末のカンマは必須
- 1行は80文字以内
- 関数の引数は3つまで
- コメントは英語で書く
  [50行のスタイルルール...]
```

### ✅ 良い例

```markdown
## コードスタイル

- Biome でフォーマット（`pnpm lint:fix`）
- 既存パターンに従う: `src/components/Button.tsx` を参照
```

**理由**: スタイルチェックは linter/formatter に任せる。LLMにさせるのは非効率。

---

## 3. 矛盾する指示

### ❌ 悪い例

```markdown
## コード規約

- 常に関数型で書くこと
- クラスベースのコンポーネントを使用すること
```

### ✅ 良い例

```markdown
## コード規約

- 関数型コンポーネントを使用（クラスコンポーネントは使わない）
```

**理由**: 矛盾する指示があるとAIは予測不能な動作をする

---

## 4. 変わりやすい情報の記載

### ❌ 悪い例

```markdown
## 現在の状況

- 現在のエントリーポイントは src/index.ts
- 本番環境URL: https://api.example.com
- 現在のバージョン: 2.3.1
- 最終更新: 2024年1月15日
```

### ✅ 良い例

```markdown
## エントリーポイント

- package.json の main フィールドを参照
```

**理由**: 変わりやすい情報はすぐに陳腐化し、コンテキストを汚染する

---

## 5. 指示形式の乱用

### ❌ 悪い例

```markdown
## ルール

- 必ず〜してください
- 絶対に〜しないでください
- 常に〜を確認してください
- 〜は禁止です
  [20個の命令文...]
```

### ✅ 良い例

```markdown
## プロジェクトの特徴

- TypeScript strict mode を使用
- テストは Vitest で実行
- API通信には React Query を使用
```

**理由**: 指示が多すぎると全体が「ノイズ」として処理される。事実形式の方が無視されにくい。

---

## 6. /init や自動生成の内容をそのまま使う

### ❌ 悪い例

```markdown
# AGENTS.md

This is a Next.js project bootstrapped with create-next-app.

## Getting Started

First, run the development server:

npm run dev

# or

yarn dev

# or

pnpm dev
```

**理由**: 自動生成は余計な情報が多い。手動で必要最小限に絞り込む。

## 7. README.md の内容をコピー

### ❌ 悪い例

```markdown
# AGENTS.md

[README.md の内容をそのままコピー]

## Installation

## Usage

## Contributing

## License
```

### ✅ 良い例

```markdown
# AGENTS.md

## プロジェクト概要

[AIエージェント向けに最適化した簡潔な説明]

## コマンド

[開発に必要なコマンドのみ]
```

**理由**: README.md は人間向け、AGENTS.md はAIエージェント向け。目的が異なる。

## 8. コードスニペットの埋め込み

### ❌ 悪い例

```markdown
## APIレスポンス形式

\`\`\`json
{
"status": "success",
"data": {
"id": 1,
"name": "example",
"created_at": "2024-01-01T00:00:00Z"
}
}
\`\`\`
```

### ✅ 良い例

```markdown
## APIレスポンス形式

- src/types/api.ts の Response 型を参照
```

**理由**: コードスニペットは陳腐化しやすい。`file:line` 参照で一次情報を指す。

---

## 9. すべてのケースを網羅しようとする

### ❌ 悪い例

```markdown
## エラーハンドリング

- 400エラーの場合: ...
- 401エラーの場合: ...
- 403エラーの場合: ...
- 404エラーの場合: ...
- 500エラーの場合: ...
  [全HTTPステータスコードの説明...]
```

### ✅ 良い例

```markdown
## エラーハンドリング

- src/utils/error-handler.ts のパターンに従う
```

**理由**: 網羅的な記述はコンテキストを圧迫する。代表的なパターンへの参照で十分。

## 10. 稀なケース向けの指示

### ❌ 悪い例

```markdown
## 特殊ケース

- レガシーブラウザ対応時は polyfill を追加
- IE11対応時は以下の設定を...
- Windows環境でのパス区切り文字について...
```

### ✅ 良い例

（そもそも記載しない）

**理由**: 滅多に発生しないケースは記載不要。必要になったときにプロンプトで伝える。
