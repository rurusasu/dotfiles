# 08 - Messages 設定

OpenClaw のメッセージ処理パイプラインに関する設定。
受信メッセージのキューイング、デバウンス、リアクション制御を行う。

## 現在の設定値

### ackReactionScope

| キー               | 値                 | 説明                                         |
| ------------------ | ------------------ | -------------------------------------------- |
| `ackReactionScope` | `"group-mentions"` | グループメンションに対してのみリアクション絵文字を付与 |

### queue

メッセージキューの動作を制御する。

| キー             | 値            | 説明                                       |
| ---------------- | ------------- | ------------------------------------------ |
| `queue.mode`     | `"collect"`   | 複数メッセージを収集してからまとめて処理   |
| `queue.debounceMs` | `1500`      | 次のメッセージを 1500ms 待機してから処理開始 |
| `queue.cap`      | `20`          | キューに保持する最大メッセージ数           |
| `queue.drop`     | `"summarize"` | キュー溢れ時、破棄メッセージを要約して保持 |

### inbound

受信メッセージのデバウンス設定。

| キー                  | 値    | 説明                                 |
| --------------------- | ----- | ------------------------------------ |
| `inbound.debounceMs`  | `800` | 受信メッセージに対する 800ms デバウンス |

## 設計判断

### collect モードの採用

ユーザーが短いメッセージを連続送信するケースに対応するため、`collect` モードを採用している。
これにより複数メッセージが 1 回のエージェントターンにバッチ処理され、以下の利点がある:

- エージェントの呼び出し回数を削減しコストを抑制
- 文脈が分断されず一貫した応答を生成可能

### debounceMs: 1500

1500ms は「短文を複数回に分けて送るユーザー」の入力パターンに最適化した値。
短すぎると分割送信を拾えず、長すぎると応答遅延が目立つ。

### cap: 20

キュー上限を 20 に設定し、キューの無制限な成長を防止する。
通常の会話フローでは 20 メッセージを超えることは稀であり、十分なバッファとなる。

### drop: "summarize"

キュー溢れ時に単純破棄ではなく要約を生成することで、ユーザーの意図を可能な限り保持する。

### inbound debounceMs: 800

Webhook や API からの重複受信を防止する。
プラットフォーム側のリトライやネットワーク遅延による二重処理を回避するための安全マージン。
